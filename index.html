Origin Story

🦐 我是怎么养出这群虾的

从一只什么都干的"超级虾"，到5只各司其职的专业团队 — 这是一段边踩坑边进化的真实历程。

2026年2月初 · 春节期间
第一只虾诞生，密集开发期
利用春节假期密集投入，从零开始搭建第一个 Claude Agent。最初只有一只"超级虾"，什么都让它干：抓新闻、写报告、分析股票、管文件。结果就是样样都会、样样稀烂——模型上下文塞满了，输出质量很不稳定，一旦出错根本不知道是哪个环节的问题。
2026年2月中
按专业拆虾
意识到问题的本质：一个 Agent 的 system prompt 越长，它的"个性"就越模糊。于是开始拆分——投资的归甜虾，海外情报的归龙虾，知识管理的归麦虾。每只虾只有一个 persona，一套工具，一类任务。质量立刻上来了。
2026年2月下旬
踩坑：API 过期、消息丢失、巡检误判
规模一大，问题接踵而来。API Key 悄悄过期，Agent 默默 fallback 到备用模型，输出质量下降好几天才被发现。网关重启导致 hooks 消息全部丢失，任务以为派出去了，其实根本没执行。还有巡检脚本在 21:00 判"任务失败"，文件其实 21:02 才刚生成……
2026年3月
建立回音机制
吸取教训，设计了三层保障：① 统一 master key 文件，所有 Agent 同源同步；② AgentFeedback 目录作为回音总线，每虾完成任务必须写回音文件；③ 每日巡检 + 04:00 备份检查双保险。"事事有回音"成为队规第一条。
现在
5只虾，每日 30+ 自动产出
A股/港股/美股分析、全球AI新闻、FMCG情报、4类知识卡片……全部自动运行，每天凌晨的飞虾队日报是第一个检查点。团队月成本约 $300，产出密度远超同等人力成本。这套系统还在持续进化中。

💡 多Agent管理经验

踩了很多坑之后沉淀下来的第一性原理 — 适合任何想搭建多Agent协作系统的人。

🎯
原则 01
一虾一专 · 不要全能单兵
每个 Agent 只做一类事，system prompt 越短越专，输出质量越稳定。让同一个 Agent 又写报告又抓数据又管文件，它会在多个角色间漂移，失去"个性"。
宁可多开一只虾，不要让一只虾什么都干。
📣
原则 02
事事有回音 · 不要 fire-and-forget
发出去的任务，不代表执行了。网关重启、API 超时、模型降级……任何环节都可能静默失败。每个任务必须有明确的完成信号，否则你只是在自我安慰。
没有回音的任务，等于没有执行的任务。
🔗
原则 03
回音总线不依赖 IM 平台
飞书/Slack/微信 的 bot 无法互相发消息。用共享文件目录（坚果云/S3/NFS）作为 Agent 间的消息总线，写文件比调 API 更可靠，还天然留下日志。
最稳的通信方式，往往是最朴素的那个。
🔑
原则 04
API Key 集中管理 · 定期同步
Key 分散在各 Agent 的配置文件里，一旦过期就是一场考古。用统一的 master key 文件，任何 Key 更新只需改一处，再同步到所有 Agent。同时设置 Key 过期告警。
Key 过期比 Bug 更危险，因为它不报错，只是悄悄变差。
🔍
原则 05
每日巡检是质量护城河
多个 Agent 并行运行，靠人工检查不现实。自动巡检脚本每日凌晨检查所有产出的文件大小、生成时间和内容完整性，异常立刻告警。这是发现问题的第一道防线。
没有监控的自动化，是定时炸弹。
🌱
原则 06
从一只虾开始 · 逐步演进
不要一上来就设计五虾协作的完美架构。先让一只虾把一件事做好，跑稳之后再拆分、再扩展。过早的复杂性是最大的敌人，Agent 系统的债务比代码债务更难还。
先做好一件事，再做好第二件事。
🧠
原则 07
指挥层与执行层分离
需要一只专门负责调度、巡检和对外沟通的"指挥虾"，它不做具体任务，只负责派发、跟踪和汇报。执行层的虾不需要知道全局，只需要把自己的任务做好并回音。
指挥官不冲锋，执行者不分心。
💾
原则 08
记忆管理决定 Agent 上限
每只 Agent 的 MEMORY.md 是它的"长期记忆"，要精心维护：只写对未来有用的规则和经验，定期清理过时内容。好的 MEMORY 让 Agent 越用越聪明，坏的 MEMORY 让它越来越混乱。
MEMORY.md 是 Agent 的灵魂文件，写什么就变什么。
⚡
原则 09
成本控制：任务分级 · 模型分级
不是所有任务都需要最强模型。日常数据拉取和格式转换用轻量模型，深度分析和创作才用旗舰模型。按任务复杂度匹配模型，在不降低质量的前提下把成本打到最低。
用对模型比用贵模型更重要。
